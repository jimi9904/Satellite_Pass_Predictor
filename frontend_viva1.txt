FRONTEND TECH STACK
================================================================================

The frontend of this project uses the following technologies:

React 19.2.0 - The main JavaScript library for building user interfaces. React allows us to create reusable components and manage application state efficiently.

Vite 7.2.2 - The build tool and development server. Vite is much faster than Create React App because it uses native ES modules and provides instant server start and hot module replacement during development.

Tailwind CSS 3.4.14 - A utility-first CSS framework. Instead of writing custom CSS files, we use Tailwind utility classes directly in JSX. This makes styling faster and keeps styles consistent across the application.

Framer Motion 12.23.25 - A powerful animation library for React. It provides smooth animations for page transitions, hover effects, and component animations. All animations in this project use Framer Motion.

React Router DOM 7.10.0 - Handles client-side routing. It allows navigation between pages without full page reloads, making the application feel like a single-page application.

Axios 1.13.2 - HTTP client library for making API requests to the backend. It handles request and response interceptors, automatic JSON parsing, and error handling.

React Leaflet 5.0.0 - React wrapper for Leaflet maps. Used in MapView.jsx to display interactive maps with satellite positions, orbit paths, and footprints.

React i18next 16.4.1 - Internationalization library. Enables the application to support multiple languages. Currently supports English and Hindi, with translations stored in JSON files.

React Icons 5.5.0 - Icon library providing HeroIcons and FontAwesome icons used throughout the UI for buttons, navigation, and visual indicators.

All these dependencies are defined in frontend/package.json.

ROUTING SYSTEM
================================================================================

The routing system uses React Router DOM and is configured in App.jsx.

PUBLIC ROUTES (No authentication required):
- /login - Login page
- /signup - Registration page

These routes are accessible to everyone, even without logging in.

PROTECTED ROUTES (Require authentication):
All other routes are wrapped in PrivateRoute component:
- / - Home page with pass prediction
- /satellites - List of all satellites
- /satellites/:id - Individual satellite details page
- /map - Real-time satellite tracking map
- /disaster-support - Disaster management information
- /self-reliance - Atmanirbhar Bharat content
- /learn - Learning hub main page
- /learn/orbits - Orbits tutorial
- /learn/launchers - Launch vehicles information
- /learn/gaganyaan - Gaganyaan mission details
- /learn/history - ISRO history
- /about - About ISRO page
- /feedback - Feedback submission form

HOW ROUTING WORKS:

1. BrowserRouter is set up in main.jsx, wrapping the entire application
2. App.jsx uses Routes component to define all routes
3. Each protected route is wrapped in PrivateRoute component
4. PrivateRoute checks if user is authenticated using useAuth hook
5. If not authenticated, PrivateRoute redirects to /login using Navigate component
6. If authenticated, PrivateRoute renders the requested page component

PAGE TRANSITIONS:
- AnimatePresence from Framer Motion wraps the Routes
- Each route change triggers exit and enter animations
- Pages fade in with a slight upward motion (opacity 0 to 1, y: 20 to 0)
- Spring animation provides smooth, natural motion

ROUTE PARAMETERS:
- Dynamic routes like /satellites/:id use useParams hook to extract the id
- Example: In SatelliteDetails.jsx, const { id } = useParams() gets the satellite ID from URL


AUTHENTICATION FLOW
================================================================================

JWT HANDLING:

The application uses JSON Web Tokens (JWT) for authentication. Here's how it works:

1. When user logs in or signs up, backend returns a JWT token
2. Token is stored in browser's localStorage with key "token"
3. Token is also set in Axios default headers: Authorization: Bearer <token>
4. Every API request automatically includes this token in headers
5. Backend verifies token on protected endpoints
6. If token is invalid or expired, backend returns 401, frontend redirects to login

LOGIN FLOW:

1. User enters email and password in Login.jsx form
2. Form submission calls handleSubmit function
3. handleSubmit calls AuthContext.login(email, password)
4. AuthContext makes POST request to /api/auth/login with credentials
5. Backend validates credentials and returns JWT token and user data
6. Frontend stores token: localStorage.setItem("token", token)
7. Frontend sets token in Axios headers: api.defaults.headers.common["Authorization"] = `Bearer ${token}`
8. Frontend updates user state: setUser(res.data.user)
9. User is redirected to home page: navigate("/", { replace: true })

SIGNUP FLOW:

1. User enters name, email, and password in Signup.jsx form
2. Form submission calls handleSubmit function
3. handleSubmit calls AuthContext.signup(name, email, password)
4. AuthContext makes POST request to /api/auth/signup
5. Backend validates input, hashes password, creates user, returns JWT token
6. Frontend stores token and updates state (same as login)
7. User is redirected to home page

LOGOUT FLOW:

1. User clicks logout button in NavBar
2. Calls AuthContext.logout() function
3. logout() removes token from localStorage: localStorage.removeItem("token")
4. logout() clears user state: setUser(null), setToken(null)
5. logout() removes Authorization header: delete api.defaults.headers.common["Authorization"]
6. User is redirected to login page: navigate("/login")

TOKEN PERSISTENCE:

- Token survives page refresh because it's stored in localStorage
- On page load, AuthContext checks localStorage for existing token
- If token exists, it's loaded into state and set in Axios headers
- AuthContext calls /api/auth/me to verify token and load user data
- If token is invalid, it's removed and user must login again


STATE MANAGEMENT
================================================================================

CONTEXT API:

The project uses React Context API for global state management. There is only one context: AuthContext.

AuthContext (context/AuthContext.jsx) provides:
- user: Current logged-in user object (id, email, name)
- token: JWT token string
- loading: Boolean indicating if authentication check is in progress
- isAuthenticated: Computed boolean (!!user)
- login: Function to login user
- signup: Function to register new user
- logout: Function to logout user

Why Context API instead of Redux:
- Only authentication state needs to be global
- Redux would be overkill for a single piece of global state
- Context API is built into React, no additional dependencies
- Simpler to understand and maintain

HOOKS USED:

useState - For local component state:
- Home.jsx: coords, satellites, selected, prediction, loading, error
- MapView.jsx: satellites, selected, position, orbitPath, trail, error
- Login.jsx: form (email, password), error
- Each component manages its own local state

useEffect - For side effects:
- Fetching data from API when component mounts
- Setting up intervals for real-time updates
- Cleaning up intervals when component unmounts
- Example: MapView.jsx uses setInterval to fetch position every 5 seconds

useContext - To access global state:
- useAuth() hook is a wrapper around useContext(AuthContext)
- Components call const { isAuthenticated, user, logout } = useAuth()
- This gives them access to authentication state and functions

useParams - To get route parameters:
- SatelliteDetails.jsx: const { id } = useParams() to get satellite ID from URL

useNavigate - For programmatic navigation:
- Login.jsx: navigate("/") after successful login
- NavBar.jsx: navigate("/login") after logout

useMemo - For performance optimization:
- MapView.jsx: const coordinates = useMemo(() => orbitPath.map(...), [orbitPath])
- Prevents recalculating coordinates on every render, only when orbitPath changes

DATA FLOW:

1. Global state: AuthContext provides authentication state to all components
2. API data: Components fetch data in useEffect, store in local useState, render in JSX
3. Form data: Controlled inputs update local state, submitted to API on form submit
4. Real-time updates: setInterval in useEffect fetches new data periodically


API COMMUNICATION
================================================================================

AXIOS SETUP:

The project uses Axios for all API communication. Configuration is in services/api.js:

- Base URL: Set from VITE_API_BASE_URL environment variable, defaults to http://localhost:5000/api
- Timeout: 15000ms (15 seconds)
- Response interceptor: Logs errors to console
- Request interceptor: Automatically adds Authorization header if token exists

AUTHENTICATION HEADERS:

When user logs in, token is stored and automatically added to all requests:
- Token stored in localStorage
- Token loaded into AuthContext state
- useEffect in AuthContext sets: api.defaults.headers.common["Authorization"] = `Bearer ${token}`
- Every subsequent API call includes this header automatically

API ENDPOINTS USED:

GET /api/satellites - Fetch all satellites (used in Home.jsx, SatelliteList.jsx)
GET /api/satellites/:id - Get satellite details (used in SatelliteDetails.jsx)
POST /api/predict - Predict satellite pass (used in Home.jsx, SatelliteDetails.jsx)
GET /api/position/:satId - Get real-time position (used in MapView.jsx)
POST /api/feedback - Submit feedback (used in Feedback.jsx)
POST /api/auth/login - User login (used in Login.jsx)
POST /api/auth/signup - User registration (used in Signup.jsx)
GET /api/auth/me - Verify token and get user (used in AuthContext.jsx)

ERROR HANDLING:

- All API calls wrapped in try-catch blocks
- Errors caught and stored in error state
- Error messages displayed to user in red text
- Network errors show generic error message
- 401 errors trigger logout and redirect to login

HOW IMAGES ARE HANDLED
================================================================================

SATELLITE IMAGES:

1. Image Source:
   - Images come from MongoDB database
   - Each satellite document has image_url field (string containing URL)
   - URLs can point to external CDN, Wikipedia, or any image hosting service
   - Images are not stored in database, only URLs are stored

2. Image Rendering:
   - In SatelliteDetails.jsx, image is displayed using standard img tag
   - Source: src={satellite.image_url || fallbackSatellite}
   - If image_url exists, it uses that URL
   - If image_url is null or undefined, it uses fallback image

3. Fallback Handling:
   - Fallback image is a local SVG file: fallback-satellite.svg
   - onError handler catches failed image loads
   - If image fails to load (404, network error, etc.), handler automatically switches to fallback
   - Code: onError={(e) => { e.target.onerror = null; e.target.src = fallbackSatellite; }}
   - This prevents broken image icons from appearing

4. Image Styling:
   - Full-width header image in SatelliteDetails page
   - Rounded corners: rounded-2xl
   - Object-fit: cover (maintains aspect ratio, crops if needed)
   - Gradient overlay on top for text readability
   - Height: h-64 on mobile, h-80 on desktop

5. No Image Optimization:
   - Images are loaded directly from URLs
   - Browser handles caching automatically
   - No manual image compression or lazy loading implemented
   - Could be optimized further with next/image or similar in future

LOGO AND ICONS:

- Logo: Custom SVG component (OrbitLogo) in NavBar.jsx
- Icons: React Icons library (HeroIcons and FontAwesome)
- All icons are vector-based, scale perfectly at any size
- Icons use currentColor to inherit text color

HOW ROLE/LOGIN BASED UI IS HANDLED
================================================================================

The application uses authentication-based UI, not role-based. There are only two states: authenticated and unauthenticated.

AUTHENTICATED STATE:

When user is logged in (isAuthenticated = true):

1. Navigation Bar:
   - Shows full navigation menu with all links
   - Shows logout button
   - Hides login/signup buttons

2. Route Access:
   - All protected routes are accessible
   - PrivateRoute allows access to Home, Satellites, Map, etc.
   - User can navigate freely between pages

3. API Access:
   - All API requests include Authorization header
   - Backend verifies token and returns data
   - User can fetch satellites, make predictions, submit feedback

4. UI Elements:
   - No restrictions on any UI elements
   - All features available to all authenticated users
   - No admin panels or special user types

UNAUTHENTICATED STATE:

When user is not logged in (isAuthenticated = false):

1. Navigation Bar:
   - Hides navigation menu
   - Shows login and signup buttons
   - Logo still visible and clickable

2. Route Access:
   - Only /login and /signup routes accessible
   - Attempting to access protected routes redirects to /login
   - PrivateRoute component handles this redirection

3. API Access:
   - No Authorization header in requests
   - Protected endpoints return 401 Unauthorized
   - Only public endpoints (like /api/health) work

4. UI Elements:
   - Login and signup forms are visible
   - All other content is hidden behind authentication

IMPLEMENTATION:

The authentication state is managed globally in AuthContext:
- isAuthenticated is computed as !!user (true if user exists)
- All components can check authentication using useAuth() hook
- NavBar checks isAuthenticated to show/hide menu items
- PrivateRoute checks isAuthenticated to allow/block route access

NO ROLE-BASED SYSTEM:

- There are no user roles (admin, user, guest, etc.)
- All authenticated users have the same permissions
- No role checking in frontend or backend
- Could be extended in future to add admin roles

CONDITIONAL RENDERING:

Components use conditional rendering based on authentication:
- {isAuthenticated && <NavigationMenu />}
- {!isAuthenticated && <LoginSignupButtons />}
- PrivateRoute: if (!isAuthenticated) return <Navigate to="/login" />

This ensures users only see UI elements they have access to.


WHY REACT WAS CHOSEN
================================================================================

React was chosen for this project for several important reasons:

1. Component Reusability:
   - The project has many reusable UI components (Card, Button, StatCard, SatelliteCard)
   - React's component model makes it easy to create these once and reuse across pages
   - Reduces code duplication and ensures consistency

2. State Management:
   - React hooks (useState, useEffect) handle local component state efficiently
   - Context API provides simple global state management for authentication
   - No need for complex state management libraries like Redux

3. Real-time Updates:
   - MapView page needs to update satellite positions every 5 seconds
   - React's re-rendering mechanism handles these updates smoothly
   - setInterval in useEffect triggers re-renders with new data

4. Large Ecosystem:
   - React has excellent libraries for all project needs:
     * React Leaflet for maps
     * Framer Motion for animations
     * React Router for navigation
     * React i18next for translations
   - All these libraries integrate seamlessly with React

5. Developer Experience:
   - React's declarative syntax makes code readable and maintainable
   - JSX allows writing HTML-like code in JavaScript
   - Hot module replacement in Vite provides instant feedback during development
   - Great debugging tools (React DevTools)

6. Performance:
   - Virtual DOM efficiently updates only changed components
   - React's reconciliation algorithm minimizes DOM manipulations
   - Code splitting reduces initial bundle size
   - Optimizations like useMemo prevent unnecessary recalculations

7. Industry Standard:
   - React is widely used and well-documented
   - Large community and extensive resources
   - Easy to find developers familiar with React
   - Long-term support and active development

8. Single Page Application:
   - React Router enables SPA behavior (no page reloads)
   - Smooth transitions between pages
   - Better user experience than traditional multi-page applications
